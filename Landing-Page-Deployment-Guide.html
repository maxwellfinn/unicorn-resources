üåê DNS & Domain Access Issues

‚ùå "DNS_PROBE_FINISHED_NXDOMAIN" ‚è±Ô∏è Quick Fix

The subdomain doesn't exist yet - DNS hasn't propagated or CNAME wasn't added.

‚úÖ Wait 15-30 minutes. If still not working, verify the CNAME record values are exactly correct (no typos, no trailing dots).

‚ùå Client doesn't know their DNS login

They can't access their domain registrar or DNS provider account.

‚úÖ Ask who set up their domain originally. Check email for "domain registration" receipts. Common providers: GoDaddy, Namecheap, Google Domains, Cloudflare. Try password reset.

‚ùå Domain registrar ‚â† DNS host ‚ö° Common Confusion

Client bought domain at GoDaddy but DNS is managed by Cloudflare (or vice versa).

‚úÖ Use whois.domaintools.com to look up the domain's nameservers. This tells you who actually manages DNS. If nameservers say "cloudflare" they need to log into Cloudflare, not GoDaddy.

‚ùå Client entered full domain instead of subdomain

They typed promo.example.com in the Name field instead of just promo.

‚úÖ The Name/Host field should ONLY contain the subdomain part. If they want promo.example.com, they enter just promo. The provider automatically appends the domain.

‚ùå Conflicting DNS record already exists

There's already an A record or another CNAME for the same subdomain.

‚úÖ Delete the existing record first, then add the new CNAME. You cannot have both an A record and CNAME for the same subdomain.

‚ùå Client wants root domain (not subdomain)

They want example.com instead of promo.example.com.

‚úÖ CNAME records cannot be used at the root domain (it's a DNS limitation). Options: Use Cloudflare (supports CNAME flattening), or use Netlify/Cloudflare's DNS directly and point nameservers there.

‚ùå Domain is expired or locked

Domain shows as expired, pending deletion, or has a registrar lock.

‚úÖ Client needs to renew their domain or contact their registrar to unlock it. Check expiration at whois.domaintools.com.
üîí SSL/HTTPS Issues

‚ùå "Your connection is not private" / SSL Error ‚è±Ô∏è Quick Fix

SSL certificate hasn't been issued yet (Netlify/Cloudflare provision these automatically).

‚úÖ Go to your hosting dashboard ‚Üí Domain settings ‚Üí Click "Verify DNS" or "Retry". Wait 5-10 minutes. SSL is auto-provisioned once DNS is verified.

‚ùå CAA record blocking certificate

Domain has a CAA record that only allows specific certificate authorities.

‚úÖ Check for CAA records in DNS. Either remove them or add the certificate authority: For Netlify add letsencrypt.org, for Cloudflare add digicert.com and letsencrypt.org.

‚ùå Mixed content warnings (page loads but shows "not secure")

Page is HTTPS but loads images/scripts via HTTP.

‚úÖ Check your HTML for any http:// URLs. Change all resource URLs to https:// or use protocol-relative URLs //example.com/image.jpg.

‚ùå Redirect loop (ERR_TOO_MANY_REDIRECTS)

Usually happens when Cloudflare's "Always Use HTTPS" conflicts with Netlify's forced HTTPS.

‚úÖ If using Cloudflare DNS with Netlify hosting: Turn OFF "Always Use HTTPS" in Cloudflare, OR set SSL mode to "Full (strict)" instead of "Flexible".

‚ùå Certificate doesn't cover subdomain

Client has SSL but it's not a wildcard - doesn't cover the new subdomain.

‚úÖ Netlify/Cloudflare Pages auto-provision individual certificates for each domain. If using client's own SSL, they need a wildcard cert (*.domain.com) or a new cert for the subdomain.
üìÅ Files & Asset Issues

‚ùå 404 Not Found ‚è±Ô∏è Quick Fix

Main file isn't named correctly or is in wrong location.

‚úÖ Main file MUST be named index.html (all lowercase). It must be in the root of your uploaded folder, not inside a subfolder.

‚ùå Case sensitivity issues (works locally, broken online)

Files named Image.PNG but referenced as image.png.

‚úÖ Linux servers (which Netlify/Cloudflare use) are case-sensitive. Image.PNG ‚â† image.png. Rename all files to lowercase and update HTML references.

‚ùå Images not loading (broken image icons)

Image paths are wrong or files weren't included in upload.

‚úÖ Check that: 1) All images are in the uploaded folder, 2) Paths are relative (e.g., images/photo.jpg), 3) No absolute paths like C:\Users\... or /Users/mac/....

‚ùå Spaces or special characters in filenames

File named hero image (1).jpg causes issues.

‚úÖ Rename files to remove spaces and special characters. Use hyphens or underscores: hero-image-1.jpg. Update all HTML references.

‚ùå Fonts not loading

Custom fonts or Google Fonts not displaying.

‚úÖ Check if Google Fonts is blocked (some corporate networks block it). Consider self-hosting fonts. Ensure font URLs use HTTPS.

‚ùå External scripts blocked

Third-party scripts (analytics, chat widgets, etc.) not working.

‚úÖ Some corporate firewalls block external scripts. Test from a non-corporate network. Ensure scripts use HTTPS. Check browser console for errors.
‚òÅÔ∏è Cloudflare-Specific Issues

‚ùå Orange cloud enabled (should be gray) ‚ö° Very Common

When pointing to Netlify, the Cloudflare proxy (orange cloud) causes SSL conflicts.

‚úÖ Click the orange cloud icon next to the CNAME record to turn it GRAY. Gray = DNS only (what we want). Orange = Proxied (causes issues with external hosts).

‚ùå SSL mode set to "Flexible"

Causes redirect loops when the destination already has SSL.

‚úÖ Go to SSL/TLS settings ‚Üí Change mode to "Full" or "Full (strict)". Never use "Flexible" when pointing to Netlify/Cloudflare Pages.

‚ùå Page Rules interfering

Existing page rules for redirects or caching affecting the new subdomain.

‚úÖ Check Page Rules in Cloudflare dashboard. Look for wildcard rules like *example.com/* that might match the new subdomain. Disable or modify as needed.

‚ùå Under Attack Mode enabled

All visitors see a Cloudflare challenge page before the site.

‚úÖ Go to Security ‚Üí Under Attack Mode ‚Üí Turn OFF (or set to specific pages only). This is meant for DDoS attacks, not normal operation.

‚ùå Caching showing old content

Cloudflare cached the old version and isn't showing updates.

‚úÖ Go to Caching ‚Üí Purge Everything. Or use Development Mode temporarily to bypass cache during testing.
üíª Client Environment Issues

‚ùå Works for you, not for client

DNS propagated to your location but not theirs yet.

‚úÖ DNS propagation varies by location. Ask client to wait 30 min and try again. Or try on mobile data (different DNS servers).

‚ùå Browser caching old DNS

Client's browser cached the "domain not found" response.

‚úÖ Have client: 1) Clear browser cache, 2) Try incognito/private window, 3) Try different browser, 4) Flush DNS cache (Windows: ipconfig /flushdns, Mac: sudo dscacheutil -flushcache).

‚ùå Corporate firewall blocking subdomain

Client's company firewall blocks new/unknown subdomains.

‚úÖ Ask client to test on their phone (not on company WiFi) or from home. If it works there, they need IT to whitelist the subdomain.

‚ùå VPN interfering

Client's VPN uses different DNS servers that haven't updated.

‚úÖ Ask client to temporarily disconnect VPN and test. VPNs often have slower DNS propagation.

‚ùå ISP DNS caching

Client's internet provider has aggressive DNS caching.

‚úÖ Try using Google DNS (8.8.8.8) or Cloudflare DNS (1.1.1.1) instead. Or wait 24-48 hours for ISP cache to expire.
üè¢ Business & Access Blockers

‚ùå "IT handles our DNS"

Client doesn't have direct access - their IT department manages everything.

‚úÖ Prepare clear instructions for IT: Provide the exact CNAME record details in a formal email. Offer to join a call with IT if needed. Set expectations for 24-48hr turnaround.
Hi IT Team, We need to add a DNS record for a new landing page: Type: CNAME Name: [subdomain] Target: [your-site].netlify.app TTL: Auto/Default Please let us know once added. Thanks!

‚ùå Multiple stakeholders need approval

Marketing can't make DNS changes without leadership/legal/IT approval.

‚úÖ Offer to provide documentation explaining what CNAME records do (they don't affect email, main site, or security). Frame it as "pointing a subdomain to a landing page" - very low risk.

‚ùå Client locked out of registrar account

Forgot password, no access to 2FA, or the person who set it up left the company.

‚úÖ Contact the domain registrar's support directly. They can help recover access with proof of ownership (usually requires business documentation). This can take days.

‚ùå Subdomain already in use

Client wants promo.site.com but it's already pointing somewhere else.

‚úÖ Options: 1) Use a different subdomain (e.g., offer.site.com, lp.site.com), 2) If old subdomain is unused, delete the old record first, 3) Ask client what the existing subdomain is used for.

‚ùå "Can you just host it for me?"

Client doesn't want to deal with any of this.

‚úÖ Switch to Path A! Host it yourself on Netlify/Cloudflare and send them the simple CNAME instructions. You maintain control, they just add one record.

‚ùå Agency or contractor manages their site

A third-party agency has all the access, not the client.

‚úÖ Get the agency's contact info. Send them professional instructions. Offer to collaborate - agencies usually understand DNS quickly. May charge the client for the change.
